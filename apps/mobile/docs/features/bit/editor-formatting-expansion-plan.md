# Bit Editor Formatting Expansion Plan

This document turns the editor-formatting research into a concrete implementation plan for the LaughTrack mobile bit editor.

It does not describe code that has already been shipped. It is a rollout plan for expanding the current bit editor beyond `H1`, `H2`, and paragraph while staying compatible with the app's current HTML storage and rendering pipeline.

## Summary

Recommended implementation priority:

1. Bold
2. Italic
3. Unordered list
4. Ordered list
5. Blockquote
6. Underline
7. Strikethrough
8. Inline code
9. Links

Defer for now:

- H3
- Code block
- Checkbox list
- Mentions
- Images
- H4-H6
- Any nested list behavior

## Why These Formats Fit LaughTrack

Bits in LaughTrack are not generic documents. They are short performance artifacts that usually benefit from:

- emphasis for stress words, act-outs, and alternate reads
- lightweight structure for setups, tags, callbacks, and closers
- quick scanning in rehearsal and setlist-reader views
- stable HTML persistence that still works when flattened into previews and search text

The top-priority styles map well to actual comedy-writing needs:

- `Bold`: stress words, key punch terms, important transitions
- `Italic`: voice shifts, internal monologue, alternate delivery, side comments
- `Unordered list`: tags, alt punchlines, callback ideas, riff buckets
- `Ordered list`: sequence-based beats, escalation ladders, story steps
- `Blockquote`: premise notes, audience observation notes, coach notes, framing text
- `Underline`: exact emphasis cue for stage delivery
- `Strikethrough`: cut line, retired wording, line under review
- `Inline code`: exact wording or "say it exactly like this" phrasing; lower value than the items above
- `Links`: references to clips, source articles, or venue notes; useful but not core to the writing flow

This rollout intentionally does not add `H3`.

Reason:

- it is technically safe in the current pipeline
- it is not necessary for joke-writing workflow
- it adds another structural choice without adding much value for short-form bit writing
- keeping the heading model at `H1` / `H2` + paragraph keeps the editor simpler and preserves the existing heading-recovery logic

## Current State

### Editor stack

- Screen: `apps/mobile/src/app/(app)/(detail)/bit/[id].tsx`
- Toolbar: `apps/mobile/src/features/bit/components/editor-toolbar.tsx`
- Editor hook: `apps/mobile/src/features/bit/hooks/use-bit-editor.ts`
- HTML helpers: `apps/mobile/src/features/bit/editor/html.ts`
- Storage: bit `content` persists as HTML in WatermelonDB

### What exists now

- Toolbar exposes only `H1`, `H2`, and paragraph
- `useBitEditor` has heading-specific handling only for `h1` and `h2`
- Content is saved as HTML generated by `react-native-enriched`
- Preview and search flatten HTML to text via `apps/mobile/src/database/utils/html.ts`
- Setlist reader sanitizes and renders saved bit HTML via:
  - `apps/mobile/src/features/setlist/reader/buildSetlistReaderHtml.ts`
  - `packages/html-utils/src/index.ts`

### Important downstream constraints

The editor cannot be treated as an isolated input. A style is only safe if it survives:

1. authoring in `react-native-enriched`
2. save/load HTML persistence
3. shared HTML sanitization
4. preview flattening for cards and list search
5. setlist reader rendering

- `Code block` is not safe today because `react-native-enriched` v0.4.x uses a custom `codeblock` representation while the sanitizer only preserves `code` and `pre`.
- `Checkbox list` is not safe today because checkbox-specific attributes are likely stripped during sanitization, turning it into a plain list.
- `Mentions` and `Images` are not safe today because their tags are not preserved by the current sanitizer.
- `H4-H6` are supported by the editor but not allowed in the current shared sanitizer.

## Setlist Reader Compatibility

The setlist reader is the most important downstream dependency for this change.

Current reader path:

1. bit HTML is loaded from `bit.content`
2. wrapper tags are removed by `stripHtmlWrapper(...)`
3. HTML is sanitized by `sanitizeHtml(...)`
4. sanitized content is inserted into `buildSetlistReaderHtml(...)`
5. content is rendered inside a `WebView`

Relevant files:

- `apps/mobile/src/app/(app)/(detail)/setlist/reader.tsx`
- `apps/mobile/src/features/setlist/reader/buildSetlistReaderHtml.ts`
- `packages/html-utils/src/index.ts`

### Safe for the current rollout

The narrowed phase-1 scope should not break the setlist reader because the reader and sanitizer already support these tags:

- bold: `b`, `strong`
- italic: `i`, `em`
- underline: `u`
- strikethrough: `s`, `strike`
- unordered list: `ul`, `li`
- ordered list: `ol`, `li`
- blockquote: `blockquote`

The reader CSS already styles all of those tags, and the sanitizer already preserves them.

### Required HTML contract for phase 1

To keep the setlist reader safe, the editor output for phase-1 formats must use semantic HTML tags already preserved by `sanitizeHtml(...)`.

Approved tag contract:

- bold -> `b` or `strong`
- italic -> `i` or `em`
- underline -> `u`
- strikethrough -> `s` or `strike`
- unordered list -> `ul` + `li`
- ordered list -> `ol` + `li`
- blockquote -> `blockquote`

Do not rely on:

- styled `span` tags
- CSS classes in saved editor output
- inline `style` attributes in saved editor output
- alternate tags not in the sanitizer allowlist such as `del` or `ins`

If any phase-1 format is emitted with unsupported tags or style-driven markup, the setlist reader probably will not crash, but the formatting will be stripped during sanitization and silently disappear.

### Reader adjustments required now

No required code changes are currently needed in the setlist reader for the narrowed phase-1 scope.

However, the implementation must treat the reader as a compatibility checkpoint and explicitly verify:

- lists render with acceptable spacing and indentation
- blockquotes remain readable and not overly muted in reader mode
- nested formatting inside lists or blockquotes still renders cleanly after sanitization
- saved content still reads well in the `WebView` after being passed through `sanitizeHtml(...)`

### Reader adjustments required later

- If `Links` are added later, update `apps/mobile/src/features/setlist/reader/buildSetlistReaderHtml.ts` to add explicit `a` styling so links do not fall back to inconsistent browser defaults, and define explicit `WebView` behavior for tapping/navigating links.
- If `Code block` is ever added, update the shared sanitizer and likely normalize editor output before the reader can safely render it.
- If `Checkbox list`, `Mentions`, or `Images` are ever added, both the sanitizer and the reader will need explicit new support.

## Rollout Strategy

### Phase 1: Safe, high-value writing formats

Target formats:

- Bold
- Italic
- Underline
- Strikethrough
- Unordered list
- Ordered list
- Blockquote

Why this phase first:

- all formats are supported by `react-native-enriched`
- all formats fit the bit-writing workflow
- all formats are either already allowed or already styled across the downstream pipeline
- they increase value without forcing an editor architecture change

### Phase 1 deliverables

- expanded toolbar in bit detail
- polished `htmlStyle` configuration for the new visible formats
- verification that cards, list previews, search, reopen-after-save, and setlist reader all behave correctly
- explicit setlist-reader compatibility verification for lists and blockquotes after sanitization

### Phase 2: Conditional, lower-priority enhancements

Target formats:

- Inline code
- Links

Why phase 2:

- both are supported by the editor and can survive storage
- both are lower product value than structure/emphasis tools
- links need extra UX decisions around insertion/edit/remove flow
- inline code needs a deliberate product meaning in a comedy-writing app so it does not confuse users

### Phase 2 deliverables

- explicit insertion/editing UX for links
- final decision on whether inline code is exposed as a power-user tool or kept out of the main toolbar
- reader styling review for anchors so linked text does not look accidental or broken
- explicit link-handling decision for the setlist reader `WebView`

### Deferred Phase: Not recommended until pipeline changes land

Deferred formats:

- Code block
- Checkbox list
- Mentions
- Images
- H4-H6

These require HTML sanitizer updates, reader updates, preview/search handling updates, or product-scope clarification before they should be exposed.

## UX Plan

### Toolbar principles

- Keep paragraph as the default writing mode.
- Keep the toolbar mobile-first and fast to scan.
- Prefer one horizontally scrollable row or two compact grouped rows over a dense grid.
- Separate structure controls from emphasis controls.
- Preserve current `H1` / `H2` access because they already exist in the writing model.
- Do not introduce new heading levels in this rollout.

### Recommended toolbar grouping

Suggested order:

1. `H1`
2. `H2`
3. `P`
4. divider
5. `B`
6. `I`
7. `U`
8. `S`
9. divider
10. bullet list
11. numbered list
12. quote

Phase 2 additions, if approved later:

- link
- inline code

### Interaction rules

- `P` should clear active heading styles.
- Heading buttons should be mutually exclusive.
- List buttons should rely on `stylesState` to reflect active state and disabled state.
- Blockquote should be represented as a paragraph-style block toggle, not an inline style.
- Toolbar buttons should use `accessibilityRole="button"` and meaningful accessibility labels.

## Detailed File-by-File Implementation Plan

### 1. `apps/mobile/src/features/bit/components/editor-toolbar.tsx`

### Required changes

- Expand the toolbar from three buttons to the approved phase-1 set.
- Add active-state rendering for:
  - `bold`
  - `italic`
  - `underline`
  - `strikeThrough`
  - `unorderedList`
  - `orderedList`
  - `blockQuote`
- Add disabled-state handling using `isBlocking` where appropriate.
- Replace the current tightly-coupled `paragraph` logic with logic that only clears heading states, not unrelated styles.

### Suggested refactor

- Move from hard-coded three-button layout to a small button config array.
- Distinguish button kinds:
  - inline style toggle
  - paragraph/block style toggle
  - paragraph reset
- Keep the toolbar implementation simple and data-driven so later phase-2 additions do not create condition-heavy JSX.

### Notes

- The current component derives paragraph state as "not H1 and not H2". That remains valid in the narrowed rollout because no new heading level is being added.
- Use `onPress` instead of `onPressIn` if accidental toggles become more common with a larger control surface.
- When adding button actions, prefer editor commands whose output maps to the semantic tag contract in this plan.

### 2. `apps/mobile/src/features/bit/hooks/use-bit-editor.ts`

### Required changes

- No heading-model change is required for the narrowed rollout.
- Existing `H1` / `H2` recovery logic should remain untouched unless a regression is found while testing the expanded toolbar.

### Why this matters

The current hook contains heading-specific recovery logic that tries to repair bad transitions after pressing Enter inside a heading. Because the rollout is no longer adding `H3`, this logic should stay stable and should not be expanded unnecessarily.

### Validation focus

- Press Enter inside `H1` and `H2`.
- Verify that the editor creates the expected new paragraph behavior.
- Verify no unexpected heading expansion across multiple blocks.

### Likely no-change area

- Inline styles should not require recovery logic changes unless a bug is discovered during manual testing.

### 3. `apps/mobile/src/features/bit/editor/html.ts`

### Required changes

- No new heading helper logic is required for the narrowed rollout.
- Keep helper behavior aligned with the actual block types exposed in the toolbar.

### Current status

- Several helpers already support `h1-h6` generically via regex.
- Heading-specific runtime state still depends on the hook-level `ActiveHeading` type and usage.

### What to confirm during implementation

- `extractTextFromHtml` still produces correct flattened text when new phase-1 styles are introduced.
- `countLogicalBreaks` continues to behave correctly for lists and blockquotes.
- paragraph detection still works when existing headings are followed by list or quote content.

### 4. `apps/mobile/src/app/(app)/(detail)/bit/[id].tsx`

### Required changes

- Extend the `htmlStyle` config to explicitly define the visible appearance of the new phase-1 formats.

### Recommended `htmlStyle` additions

- `blockquote`

- `ul`
  - balanced indent and gap sizing for mobile width
- `ol`
  - readable marker spacing and weight
- inline emphasis styles

### Additional UI review

- A larger toolbar may require extra bottom padding for the editor area.
- Confirm the toolbar does not cover the last line while typing.
- Confirm the chosen editor commands emit semantic tags preserved by the sanitizer, not style-driven markup.

## Implementation Sequence

Recommended sequence for actual engineering work:

1. Expand toolbar config and active-state handling in `editor-toolbar.tsx`
2. Keep heading logic unchanged in `use-bit-editor.ts` unless testing reveals a regression
3. Add phase-1 `htmlStyle` definitions in bit detail screen
4. Manually verify save/reopen behavior for each new format
5. Verify card preview and search behavior
6. Verify setlist-reader rendering and sanitization
7. If needed, make CSS-only spacing adjustments in the setlist reader
8. Polish spacing, accessibility labels, and button ordering

## Acceptance Criteria

Phase 1 is complete when all of the following are true:

- Users can apply `Bold`, `Italic`, `Underline`, `Strikethrough`, `H3`, `Unordered List`, `Ordered List`, and `Blockquote` from the bit editor toolbar.
- Users can apply `Bold`, `Italic`, `Underline`, `Strikethrough`, `Unordered List`, `Ordered List`, and `Blockquote` from the bit editor toolbar.
- Saving and reopening a bit preserves those formats.
- `P` returns heading content to paragraph mode cleanly.
- `H1` and `H2` still behave correctly when pressing Enter.
- Bit cards still show readable previews.
- Bit list search still matches text inside formatted content.
- Setlist reader renders the new formats cleanly after sanitization.
- No newly exposed format is silently stripped by the current pipeline.
- No setlist-reader code path regresses when formatted bits are rendered in the `WebView`.
- Toolbar remains usable on small screens and does not hide editor content.

## Risks and Mitigations

### Risk: heading behavior regression

The current hook has custom recovery logic for heading transitions.

Mitigation:

- do not expand the heading model in this rollout
- preserve the existing `H1` / `H2` logic and test it after the toolbar expands

### Risk: toolbar becomes crowded

Adding too many buttons at once could reduce usability.

Mitigation:

- implement only phase-1 buttons initially
- keep ordering semantic and compact
- use a horizontally scrollable row if needed

### Risk: HTML survives editing but looks poor in reader mode

Mitigation:

- explicitly verify reader HTML output and only tune CSS where needed

### Risk: lower-priority formats create pipeline debt

Mitigation:

- defer codeblock, checkbox, mentions, images, and deeper headings until the sanitizer and preview path are deliberately updated

## Defaults to Use Unless Product Direction Changes

- Keep `H1`, `H2`, and paragraph exactly as existing top-level structure tools.
- Treat lists and blockquote as core writing tools, not advanced options.
- Treat inline code and links as secondary tools.
- Keep `H3` out of the current rollout.
- Do not expose codeblock, checkbox list, mentions, images, or `H4-H6` in the initial rollout.

## Out of Scope

This plan does not include:

- replacing `react-native-enriched`
- moving to a JSON block editor model
- redesigning bit cards or setlist-reader UX beyond necessary formatting support
- adding remote sync semantics for richer editor content
- building a generic Notion-style editor inside the bit screen

## Deliverable Recommendation

Implement phase 1 only in the first pass.

That gives LaughTrack the highest-value writing improvements with the lowest pipeline risk and keeps the editor aligned with the actual needs of bit writing rather than turning it into a generic document editor.
